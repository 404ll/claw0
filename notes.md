## 第 01 讲：Agent 循环

**核心问题：** 怎么让 LLM 从"一问一答"变成"持续对话"？

### 思维导图

```
Agent 循环
├── 状态
│   └── messages[]：唯一的状态，每次 API 调用都传入完整数组
├── 决策点
│   └── stop_reason
│       ├── end_turn   → 打印文本，继续等待输入
│       ├── tool_use   → 执行工具，把结果还给 LLM（02讲）
│       └── max_tokens → 回复被截断
└── 错误处理
    └── API 报错 → messages.pop() 回滚，让用户重试
```

### 是什么

一个 `while True` 循环，每轮三步：收输入 → 调 API → 按 `stop_reason` 分支。

```
while True:
    input → messages.append(user)
         → LLM API
         → stop_reason == "end_turn"  → print → messages.append(assistant)
         → stop_reason == "tool_use"  → 02讲处理
```

### 为什么需要

单次 API 调用只能完成一轮对话。要让 LLM "记住"上下文，必须把历史消息每次都带上去——`messages[]` 就是这个历史容器，`while True` 保证对话可以持续。

### 会带来什么问题

`messages[]` 会无限增长，最终超出 LLM 的上下文窗口限制。

→ 这是第 03 讲 ContextGuard 存在的根本原因。

### 怎么优化

- 短期：压缩历史（03讲）
- 长期：持久化到磁盘，重启后可以恢复（03讲）

## 第 02 讲：工具使用

**核心问题：** LLM 只能输出文字，怎么让它执行真实的操作？

### 思维导图

```
工具使用
├── 告诉 LLM（TOOLS）
│   └── JSON Schema：工具名、描述、参数定义
├── 告诉代码（TOOL_HANDLERS）
│   └── dict[工具名 → Python 函数]
├── 执行流程
│   ├── LLM 返回 {name, input}（只是 JSON，不是真的执行）
│   ├── 代码查表 → 执行对应函数
│   └── 结果作为 user 消息返回给 LLM
└── 内层循环
    └── LLM 可能连续调用多个工具才产生最终文本
```

### 是什么

两个平行结构：`TOOLS`（JSON Schema，告诉 LLM 有什么）+ `TOOL_HANDLERS`（函数映射表，告诉代码怎么执行）。

LLM 只返回"意图"（JSON），实际执行由你的代码完成。

### 为什么需要

LLM 本身是无状态的文字预测器，无法读文件、执行命令、调接口。工具机制是 LLM 和外部世界之间的唯一桥梁。

### 会带来什么问题

1. **安全问题**：LLM 可能被诱导调用危险工具（如删文件）
2. **无限循环**：LLM 可能在工具之间反复调用不结束
3. **错误传播**：工具报错如果抛异常，LLM 无法感知并修正

### 怎么优化

- 工具错误返回字符串而非抛异常 → LLM 能看到错误自行修正
- 设置最大迭代次数防止无限循环（09讲）
- 工具执行前做安全校验（如路径白名单）

## 第 03 讲：会话与上下文保护

**核心问题：** 对话历史会越来越长，进程重启会丢失，上下文会爆炸——怎么办？

### 思维导图

```
会话管理
├── 持久化（SessionStore）
│   ├── 格式：JSONL，每行一条记录，追加写入
│   ├── 四种记录类型：user / assistant / tool_use / tool_result
│   └── 恢复：_rebuild_history() 重建为 API 格式
└── 上下文保护（ContextGuard）
    ├── Attempt 0：正常调用
    ├── Attempt 1：裁剪过大的 tool_result
    ├── Attempt 2：compact_history（LLM 生成摘要替换旧消息）
    └── 失败：raise 异常

文件布局：
workspace/.sessions/agents/{agent_id}/sessions/{session_id}.jsonl
```

### 是什么

- **JSONL 持久化**：每条消息追加写入文件，重启后重放恢复
- **ContextGuard**：每次 API 调用前的三阶段保护，溢出时自动压缩

### 为什么需要

`messages[]` 无限增长是第 01 讲埋下的问题。两个具体场景：
1. 进程崩溃/重启 → 对话历史全丢
2. 长对话超出 token 上限 → API 报错

### 会带来什么问题

- **压缩会丢失细节**：`compact_history` 用摘要替换旧消息，细节不可逆丢失
- **摘要质量依赖 LLM**：压缩本身也要调一次 API，增加延迟和费用
- **JSONL 文件会越来越大**：没有清理机制的话磁盘会被占满

### 怎么优化

- 重要信息主动存入 MEMORY.md（06讲），不依赖对话历史
- 定期归档旧会话文件
- 压缩前先截断大 tool_result，减少压缩必要性

## 第 04 讲：通道抽象

**核心问题：** Telegram、飞书、CLI 消息格式各不相同，Agent 逻辑怎么不感知平台差异？

### 思维导图

```
通道层
├── 接入方式
│   ├── 长轮询（Telegram）：你的代码主动问"有没有新消息"
│   └── Webhook（飞书）：平台主动推送到你的服务器
├── 归一化
│   └── InboundMessage：统一格式，Agent 只看这个
│       ├── text, sender_id, channel
│       ├── peer_id（会话范围）
│       └── media[]
├── 消息合并（核心思路：等时间窗口，合并碎片）
│   ├── 媒体组合并：相同 media_group_id → 等 500ms 攒齐
│   └── 文本合并：同一用户连续发消息 → 等 1s 静默后 flush
└── 注意事项
    ├── offset 必须持久化（防止重启后重复处理）
    └── 白名单过滤（防止陌生人使用 bot）
```

### 是什么

Channel 抽象基类只有两个方法：`receive()` + `send()`。每个平台实现自己的适配器，把平台格式转成统一的 `InboundMessage`。

### 为什么需要

没有抽象层的话，Agent 逻辑里会充满 `if channel == "telegram"` 这样的分支，每新增一个平台都要改核心逻辑，维护噩梦。

> **设计模式：Adapter** — 把不兼容的接口转换成统一格式。

### 会带来什么问题

- **消息合并的等待时间是经验值**：500ms/1s 不一定适合所有场景，太短会漏，太长用户感觉卡顿
- **长轮询有延迟**：每 30s 才问一次，消息最多延迟 30s
- **Webhook 需要公网地址**：本地开发不方便

### 怎么优化

- 长轮询改 Webhook 提升实时性（需要公网服务器）
- 消息合并的等待时间可以做成可配置项

## 第 05 讲：网关与路由

**核心问题：** 一个 bot 收到消息，怎么决定交给哪个 Agent 处理？

### 思维导图

```
路由层
├── 消息流向
│   用户 → Telegram → 你的服务器 → 网关路由 → Agent → LLM → 回复
├── BindingTable（绑定表）
│   ├── Tier 1: peer_id    最具体（某个特定用户）
│   ├── Tier 2: guild_id   某个服务器/群组
│   ├── Tier 3: account_id 某个 bot 账号
│   ├── Tier 4: channel    某个平台（如所有 Telegram 消息）
│   └── Tier 5: default    兜底（全部）
└── 会话隔离（dm_scope）
    ├── main               所有人共享一个会话
    ├── per-peer           每个用户独立（最常用）
    ├── per-channel-peer   同一用户在不同平台有不同会话
    └── per-account-channel-peer  最大隔离
```

### 是什么

在 Agent 前面加一层路由，根据消息来源（平台/账号/用户）决定交给哪个 Agent，以及用哪个会话 key 做隔离。

### 为什么需要

单一 Agent 无法针对不同用户/场景切换性格和能力。路由层让同一个 bot 账号可以根据规则分发给多个 Agent。

### 会带来什么问题

- **规则冲突**：多条绑定规则可能都匹配，需要明确优先级
- **本地单人使用感知不到**：只有 Tier 5 兜底规则，路由层形同透明

### 怎么优化

目前是线性扫描匹配，规则多了性能会下降。实际使用中规则数量有限，问题不大。

> **要真正用到多 Agent**，需要：1. 服务器有公网 IP；2. 在绑定表里配置路由规则。

## 第 06 讲：智能层

**核心问题：** 怎么给 Agent 注入稳定的性格，同时让它了解"这个具体的用户"？

### 思维导图

```
智能层
├── 启动时（缓存，只跑一次）
│   ├── BootstrapLoader：加载 SOUL.md / IDENTITY.md 等
│   └── SkillsManager：扫描所有 SKILL.md，去重缓存
│       → 这就是为什么装了新 skill 需要重启
└── 每轮（动态构建）
    ├── _auto_recall(用户输入)
    │   └── TF-IDF 搜索 → 找出最相关的记忆片段
    └── build_system_prompt()：拼 8 层
        ├── Layer 1: IDENTITY.md  身份
        ├── Layer 2: SOUL.md      性格（靠前=影响强）
        ├── Layer 3: TOOLS.md     工具指南
        ├── Layer 4: 技能块
        ├── Layer 5: 记忆（常驻 + 本轮召回）
        ├── Layer 6: 引导上下文
        ├── Layer 7: 运行时上下文
        └── Layer 8: 通道提示

记忆搜索管道：
TF-IDF → 向量搜索 → 合并加权 → 时间衰减 → MMR去重
```

### 是什么

- **性格**：SOUL.md 放第 2 层，启动时加载，固定不变
- **记忆**：每轮搜索最相关片段注入第 5 层，动态变化
- **个性化** = 性格（口吻）+ 记忆（了解这个用户）

### 为什么需要

没有这层，每轮 LLM 调用都是无状态的通用助手，不知道自己是谁，也不记得用户的偏好。

### 会带来什么问题

- **系统提示词很长**：8 层叠加消耗大量 token，直接加剧上下文压力
- **记忆可能过时**：旧记忆如果没有衰减机制，可能误导 LLM
- **记忆搜索有误差**：TF-IDF 是关键词匹配，语义理解不如向量搜索

### 怎么优化

- 时间衰减：`score × e^(-decay_rate × 距今天数)`，让旧记忆自然降权
- MMR 重排序：避免注入一堆重复片段，保证多样性
- 升级到真正的 embedding API 可以提升语义搜索质量（但增加外部依赖）
- 与第03讲配合：系统提示词越大，ContextGuard 越重要

## 第 07 讲：心跳与 Cron

**核心问题：** Agent 目前只能被动响应用户，怎么让它主动发起行为？

### 思维导图

```
主动行为
├── 心跳（周期性检查）
│   ├── should_run() 前置条件
│   │   ├── HEARTBEAT.md 存在且非空
│   │   ├── 距上次运行已超过间隔
│   │   ├── 在 active_hours 范围内
│   │   └── 心跳本身没有在运行
│   └── Lane 互斥（用户永远优先）
│       ├── 用户线程：阻塞获取锁
│       └── 心跳线程：非阻塞获取，失败则跳过
└── Cron（定时任务）
    ├── at：一次性定时
    ├── every：固定间隔
    └── cron：标准 cron 表达式
    → 连续 5 次错误自动禁用
```

### 是什么

- **心跳**：后台线程定期让 Agent 自检，有内容才输出（否则回复 `HEARTBEAT_OK` 静默）
- **Cron**：从 `CRON.json` 加载定时任务，到时间自动触发 Agent

两者的区别：

| | 心跳 | Cron |
|---|---|---|
| 触发频率 | 高频持续（每隔几分钟）| 低频精确（每天9点）|
| 目的 | Agent 自检"有没有事要做" | 执行预定的具体任务 |
| 决策者 | LLM 自己判断要不要输出 | 时间到了就执行，无需判断 |
| 类比 | 你每隔一会儿看一眼手机 | 你设了一个闹钟 |

心跳最核心的价值：**把条件判断的决策权交给 LLM**，不需要硬编码 `if 超过30分钟 then 提醒`，只需在 HEARTBEAT.md 用自然语言描述，LLM 自己决定。

### 为什么需要

纯被动的 Agent 无法做到"到时间提醒你"、"每天早上生成日报"这类主动行为，必须有独立的定时触发机制。

### 会带来什么问题

- **心跳和用户输入竞争资源**：如果不做互斥，可能同时调用 LLM，消息混乱
- **心跳输出干扰用户**：心跳没有实质内容时也输出，会打断用户体验
- **Cron 任务僵尸化**：任务持续报错但没有自动停止机制

### 怎么优化

- Lane 互斥：用户阻塞获取锁，心跳非阻塞，冲突时心跳主动让步
- `HEARTBEAT_OK` 约定：没内容时静默，有内容才输出，且去重（和上次相同则不输出）
- 连续 5 次错误自动禁用 Cron 任务，防止僵尸任务消耗资源

## 第 08 讲：消息投递

**核心问题：** 消息发送可能失败，进程可能崩溃，怎么保证消息最终一定能送达？

### 思维导图

```
可靠投递
├── 原则：先写磁盘，再尝试发送
├── 原子写入（三步保证不产生半写文件）
│   ├── 写入 .tmp.{pid}.{id}.json
│   ├── os.fsync()（数据落盘）
│   └── os.replace()（原子交换）
├── 重试生命周期
│   ├── 成功 → ack()（删文件）
│   └── 失败 → fail()
│       ├── retry_count++
│       ├── 计算退避时间
│       └── retry_count >= 5 → 移入 failed/
├── 指数退避（带抖动）
│   └── [5s, 25s, 2min, 10min] ± 20%
└── 启动时恢复扫描
    └── 自动处理上次崩溃遗留的待投递条目
```

### 是什么

消息入队时先写磁盘（原子写入），后台线程负责实际发送，失败时按指数退避重试，最终失败移入 failed 目录。

### 为什么需要

消息发送有两种失败，需要分别应对：

**第一种：网络抖动（发了但没成功）**
- 场景：Telegram API 超时、限流、短暂不可用
- 解法：指数退避重试——失败后等一段时间再试，而不是立刻放弃
- 退避时间：[5s, 25s, 2min, 10min]，每次 ±20% 抖动防止雷群效应
- 超过5次放弃，移入 failed/ 目录

**第二种：进程崩溃（还没发就挂了）**
- 场景：服务器重启、程序异常退出、断电
- 解法：预写磁盘——消息入队时先写文件，发成功了才删文件
- 进程重启后扫描磁盘上还没删的文件，继续发送

这样无论中间发生什么，消息一定会被送达。这在工程上叫 **at-least-once delivery**（至少投递一次）。

类比：就像餐厅的点单系统——服务员先把订单写在纸上，再去通知厨房。就算服务员中途摔倒了，纸上的订单还在，换个人捡起来继续执行就行。如果只是口头传达，服务员一摔倒订单就丢了。

### 会带来什么问题

- **磁盘积压**：大量失败消息堆在 failed 目录，需要人工清理
- **消息延迟**：重试等待时间最长 10min，用户可能感觉消息"消失了"
- **雷群效应**：多条消息同时失败，重试时同时打过去，可能把平台 API 再次打挂

### 怎么优化

- 指数退避 + ±20% 抖动：打散重试时间，避免雷群效应
- 启动时恢复扫描：进程重启后自动续传未完成的投递
- 消息分片：按平台限制（Telegram=4096字符）分割长消息

## 第 09 讲：弹性

**核心问题：** API Key 限流、认证失败、上下文溢出——各种错误怎么自动恢复？

### 思维导图

```
三层重试洋葱
├── Layer 1：Auth 轮换（外层）
│   └── 遍历所有 Profile，跳过冷却中的
│       ├── auth/billing 失败 → 冷却 300s
│       ├── rate_limit 失败 → 冷却 120s
│       └── timeout 失败 → 冷却 60s
├── Layer 2：溢出恢复（中层）
│   └── overflow 错误 → 截断 tool_result → compact_history → 重试
└── Layer 3：工具调用循环（内层）
    └── 同第 01/02 讲的标准循环

失败分类器（classify_failure）：
异常文本 → rate_limit / auth / timeout / billing / overflow / unknown
                                                              ↓
                                                    所有 Profile 耗尽
                                                          ↓
                                                    尝试备选模型链
                                                          ↓
                                                    全部失败 → raise
```

### 是什么

把错误按类型分类，路由到对应层处理：认证/限流问题换 Key，溢出问题压缩历史，其他问题重试。

与第08讲的区别：
- **08讲**：消息投递的可靠性——回复怎么保证送达用户
- **09讲**：API 调用的可靠性——LLM 调用失败了怎么自动恢复

```
用户发消息
    ↓
[09讲保护] LLM API 调用
    ↓
Agent 生成回复
    ↓
[08讲保护] 发送给用户
```

### 为什么需要

单个 API Key 有速率限制，生产环境不能因为一个 Key 限流就整体不可用。**出了问题先分类，再按类型决定怎么恢复**——这个思维方式比代码本身更重要。

### 会带来什么问题

- **冷却时间是固定经验值**：rate_limit 冷却 120s 不一定够，不同 Key 的限流策略不同
- **备选模型能力不同**：fallback 到更小的模型，回复质量可能下降
- **分类器依赖字符串匹配**：错误信息格式变了就失效

### 怎么优化

- 按 HTTP 状态码分类比字符串匹配更可靠
- 冷却时间可以从 API 的 `Retry-After` 响应头读取
- 备选模型链按能力从高到低排列，尽量用最接近的模型

## 第 10 讲：并发

**核心问题：** 用户输入、心跳、Cron 同时触发，怎么保证不互相干扰、用户永远优先？

### 思维导图

```
命名 Lane 系统
├── Lane 结构
│   ├── FIFO deque（任务队列）
│   ├── Condition（同步原语）
│   ├── active_count（当前运行数）
│   ├── max_concurrency（并发上限，默认1）
│   └── generation（版本号，防僵尸）
├── 标准 Lane
│   ├── main      用户输入，阻塞等待结果
│   ├── cron      定时任务
│   └── heartbeat 心跳
├── 自泵送
│   └── 任务完成 → _task_done → _pump() → 拉取下一个
└── Generation 计数器
    └── 重启时所有 Lane generation++
        → 旧任务完成时 gen 不匹配 → 静默丢弃，不再泵送
```

### 是什么

每个类型的任务进入独立的命名 Lane（FIFO 队列），Lane 内串行执行（max=1），Lane 间并行。任务通过 `Future` 返回结果。

**为什么 Lane 之间并行不会冲突？**

因为每条 Lane 有自己独立的 `messages[]`，根本不存在共享状态：

```
main Lane：      用户A的 messages[] → LLM → 回复A
cron Lane：      定时任务的 messages[] → LLM → 执行任务
heartbeat Lane： 心跳的 messages[] → LLM → 自检
```

冲突只会发生在共享状态上。07讲需要锁是因为只有一个全局 `messages[]`，10讲每条 Lane 各自独立，所以不需要互相等待。

类比：两个服务员同时接待两桌客人，各自记各自的点单，不会冲突。冲突只会发生在"共用一张纸记两桌订单"的情况下。

### 为什么需要

第07讲用全局 Lock 做互斥，是"一条路排成一排"——同一时刻只有一个任务能跑，任务类型多了之后"谁该等谁"变得混乱。

10讲是对07讲的升级：把任务拆进独立 Lane，"多条车道各跑各的"，结构更清晰，新增任务类型直接加一条 Lane。

| | 07讲（Lock）| 10讲（Lane）|
|---|---|---|
| 模型 | 全局一把锁，同一时刻只有一个任务 | 每个 Lane 独立，Lane 之间并行 |
| 扩展性 | 任务类型多了难以管理 | 直接新增 Lane，互不影响 |

### 会带来什么问题

- **每个任务开一个线程**：大量任务会产生大量线程，有开销
- **Lane 惰性创建**：首次使用才创建，不方便提前配置
- **Generation 只在重启时递增**：运行时无法取消单个任务

### 怎么优化

- 用线程池替代每任务一线程（OpenClaw 生产代码已这样做）
- `wait_for_idle()` + Condition.wait() 实现高效的空闲检测，避免轮询
- Generation 机制让重启后的过期任务静默死亡，不污染新的运行状态
